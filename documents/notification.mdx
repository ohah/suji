# Notification

Electron `Notification` 호환 — macOS UNUserNotificationCenter (NSUserNotification은
deprecated 후 macOS 26 제거), Linux freedesktop notification D-Bus, Windows
`Shell_NotifyIconW` balloon notification. Frontend JS + Zig/Rust/Go/Node 백엔드 SDK 모두 노출.

## 플랫폼 지원

| API | macOS | Linux | Windows |
|---|---|---|---|
| `notification.isSupported` | ✅ Bundle ID 있을 때만 true | ✅ notification daemon probe | ✅ true (Shell_NotifyIcon balloon) |
| `notification.requestPermission` | ✅ OS 다이얼로그 | ✅ daemon reachable (no prompt) | ✅ true (권한 프롬프트 없음) |
| `notification.show` | ✅ UNUserNotificationCenter | ✅ D-Bus `Notify` | ✅ `NIF_INFO` balloon |
| `notification.close` | ✅ pending + delivered | ✅ D-Bus `CloseNotification` | ✅ backing tray icon 제거 |
| Click event | ✅ `notification:click` | — | ✅ balloon click |

Linux는 `org.freedesktop.Notifications` 서비스가 없으면 graceful false.
Windows는 WinRT XML toast가 아니라 Shell notification balloon 경로다. Windows 10+에서는
OS가 balloon을 toast UI처럼 렌더링하지만 action button/reply/custom image는 지원하지 않는다.

## ⚠️ macOS Bundle ID 필수 (dev 모드 한계)

`UNUserNotificationCenter`는 **valid Bundle ID + Info.plist** 필요. `suji dev`로
실행하면 binary가 loose라 Bundle ID 없음 → `isSupported()` false 반환 + 알림 안 뜸.

**정상 동작 조건**:
- `suji build` → `.app` 번들 생성 (Info.plist에 `CFBundleIdentifier` 자동 포함)
- 또는 `Info.plist` 수동 작성 + 코드사인

**dev 모드에서**:
- `isSupported()` → false
- `show()` → `success: false` (notification 안 뜸)
- E2E는 IPC wiring + 응답 형식만 검증. 실제 알림은 `.app` 번들 manual 검증.

## v1 한계

| 미지원 | 비고 |
|---|---|
| **Actions / 버튼** | 코어 `notification` v1에는 없음. 공식 `notification-rich` plugin은 macOS/Linux action click 라우팅을 지원하고, Windows click callback은 WinRT activator 등록 후속. |
| **Reply input** | macOS/Windows native reply는 v2 |
| **커스텀 icon** | macOS는 앱 아이콘만 자동, Windows는 backing tray icon 기본값. attachment/iconPath는 v2 |
| **커스텀 sound** | macOS default/silent. Linux `silent`는 `suppress-sound` hint, Windows는 `NIIF_NOSOUND` |

## API

```ts
import { notification } from '@suji/api';

// 플랫폼 + Bundle ID 검사
const supported = await notification.isSupported();
if (!supported) {
  console.warn('Notifications require a bundled app (.app)');
  return;
}

// 권한 요청 — 첫 호출만 OS 다이얼로그
const granted = await notification.requestPermission();
if (!granted) return;

// 알림 표시
const { notificationId, success } = await notification.show({
  title: 'Build complete',
  body: 'Your project compiled successfully.',
  silent: false,
});

// 클릭 수신
suji.on('notification:click', ({ notificationId: clickedId }) => {
  if (clickedId === notificationId) {
    // bring main window forward etc.
  }
});

// 명시 close (자동 dismiss 외에)
await notification.close(notificationId);
```

## 동작 정밀

### notification.m (block completion handler 인프라 재사용)
- `dialog.m`처럼 nested NSApp event loop 패턴으로 `requestAuthorizationWithOptions:completionHandler:`
  와 `addNotificationRequest:withCompletionHandler:`를 동기화.
- `SujiNotificationDelegate` ObjC 클래스 — `UNUserNotificationCenterDelegate` 구현:
  - `willPresentNotification:` — foreground 상태에서도 banner 표시. `silent: true`면 sound option을 요청하지 않음
  - `didReceiveNotificationResponse:` — 클릭 시 NSString identifier → C 콜백 → Zig → EventBus emit
- Bundle ID 검사 (`mainBundle.bundleIdentifier`)로 dev 모드에서 안전한 stub 동작.

### notificationId 전략
- main.zig가 단조증가 카운터로 `suji-notif-{N}` 식별자 발급 (캐릭터 안전)
- caller가 `close(notificationId)`로 명시 제거 가능
- pending(미발화) + delivered(이미 표시) 둘 다 제거

### Click 라우팅
1. 사용자 알림 클릭 → macOS가 `didReceiveNotificationResponse:` 호출
2. delegate가 `g_notification_click_callback` C 함수 포인터 호출 (notificationId C string 전달)
3. `cef_notification.zig`의 `notificationClickC`가 `g_notification_emit_handler?` 호출
4. main.zig의 `notificationEmitHandler`가 `notification:click {notificationId}` EventBus emit
5. JS/SDK 리스너 수신

### Linux D-Bus backend

- `isSupported()` / `requestPermission()`은 `org.freedesktop.Notifications.GetServerInformation`
  호출 성공 여부를 반환. freedesktop notification spec에는 권한 프롬프트가 없다.
- `show()`는 `Notify("Suji", 0, "", title, body, [], hints, -1)` 호출 후 반환된 D-Bus
  `uint32` id를 Suji `notificationId`와 매핑한다. `silent:true`면
  `hints["suppress-sound"] = true`를 전달한다.
- `close()`는 Suji `notificationId`로 저장된 D-Bus id를 찾아 `CloseNotification(uint32)` 호출.
- Linux click/action signal 라우팅, custom icon, reply, action button은 코어 v1 범위 밖.
  action button이 필요하면 공식 `notification-rich` plugin의 Freedesktop
  Notifications D-Bus action 경로를 사용한다.

### Windows Shell_NotifyIcon backend

- `isSupported()` / `requestPermission()`은 true를 반환한다. 별도 권한 프롬프트가 없다.
- `show()`는 notification 전용 tray icon을 만들고 `Shell_NotifyIconW(NIM_MODIFY)` +
  `NIF_INFO`로 title/body balloon을 표시한다. `silent:true`면 `NIIF_NOSOUND`를 사용한다.
- Suji `notificationId` ↔ backing tray id를 매핑한다. `close()`는 해당 tray icon을
  `NIM_DELETE`로 제거한다.
- 사용자가 balloon을 클릭하면 Windows pump thread가 `NIN_BALLOONUSERCLICK`을 받고
  `notification:click {notificationId}`를 emit한다.
- WinRT toast XML, AppUserModelID activator, action button, reply input은 아직 v1 범위 밖이다.

## E2E 한계

macOS OS 알림 표시 자체는 puppeteer로 검증 불가 (UNUserNotificationCenter는 process 외부).
자동화 범위:
1. **회귀 가드** (`tests/cef_ipc_test.zig`) — 정적 ObjC selector / D-Bus method / cmd / SDK 노출.
2. **응답 형식 검증** (`tests/e2e/notification.test.ts`) — IPC 응답 shape (notificationId, success, granted, supported).
3. **Linux D-Bus runtime E2E** (`tests/e2e/notification-linux-runtime.test.ts`) —
   fake `org.freedesktop.Notifications` daemon으로 `Notify`/`CloseNotification` 왕복 검증.
4. **Windows runtime 경로** — Win32 Shell_NotifyIcon source/command wiring과 Windows 빌드로
   regression을 고정한다. 실제 toast/balloon 표시와 클릭은 runner UI 한계로 수동 검증 영역이다.
5. **macOS 수동 검증 필수** — `.app` 번들 후 권한 grant 후 알림 시각/사운드 확인.
